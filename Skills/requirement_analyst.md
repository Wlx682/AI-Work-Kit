# 需求分析助手 Skill

当用户说「分析需求」「看 PRD」「需求评审」「需求有什么问题」「push 产品」「需求遗漏」时执行。

> **质量差的需求更要先走本 Skill**，再进 `feature-dev-assistant`。

## 知识库

- 清单：`Contexts/需求分析/PRD分析检查清单.md`（含**整体遗漏**维度）
- 示例模式：`Contexts/需求分析/示例-创建自动化任务-PRD问题模式.md`
- 模板：`Templates/需求分析模板.md`
- Plan：`Plans/需求分析/`
- 代码对照：**当前 Cursor 工作区**（多仓库时不写死路径；工作区是 AI-Work-Kit 时用 `仓库=` 指明）

## 新分析（七步）

1. 索要 **PRD 全文或飞书链接**（链接只用于当次，不写回 Contexts）。
2. 读检查清单；按七步输出 plan。
3. **在代码库搜索**相关模块，填「与现状对照」；**对照同类功能**找 PRD 应写但未写的环节。
4. **必须输出三块**：
   - **逻辑问题表**（写了但矛盾、未定义分支）
   - **交互冲突表**（入口、空态、创建/编辑、Tab）
   - **整体遗漏表**（PRD 全文未提及或明显不全的用户旅程、异常态、接口、删除/启停、埋点、验收等）
5. **用户旅程闭环图**（文字或 mermaid）：标出 PRD 已写 / 未写环节。
6. **待产品确认**：分「逻辑交互」与「**需求遗漏**」；P0 遗漏与 P0 矛盾同等阻塞开发。
7. Plan → `Plans/需求分析/YYYY-MM-DD-模块名.md`

## 遗漏分析要求（重点）

- 不只挑 PRD 错别字，要回答：**完整做成功能，还缺哪几段 PRD？**
- 每条遗漏写清：**缺失内容**、**不补时开发只能怎么猜**、**严重度**。
- 对照检查清单 §3.1–3.6 逐类扫一遍，无遗漏则显式写「本类 PRD 已覆盖」。
- 与代码库对照：若代码里已有类似流程（如列表刷新、删除），PRD 未写 → 标为遗漏或「应与现网一致」。

## 续做

读 plan + 产品回复 / PRD 新版本 → 更新三类问题状态 → 是否可转开发。

## 与 feature-dev-assistant 衔接

| 结论 | 下一步 |
|------|--------|
| P0（含关键遗漏）未闭环 | 继续 `/requirement-analyst` 或用户「接受风险」并记录 |
| 已闭环 | `/feature-dev-assistant 新任务`，引用需求分析 plan |

## 输出原则

- 挑剔但建设性：每条给补全建议或 2–3 个方案。
- 遗漏项标注「PRD 全文未提及」，不与逻辑矛盾混为一谈。
- Push 产品时分三块：**逻辑 · 交互 · 遗漏**。

## 触发示例

```
/requirement-analyst 新分析，PRD=【飞书链接】，模块=创建自动化任务

/requirement-analyst 续做，plan=需求分析/xxx.md，进度=产品补了删除流程，更新遗漏表
```

在**代码项目**里开 Cursor 即可，不用写 `仓库=`。仅在 AI-Work-Kit 工作区时加：`仓库=/path/to/项目`。
