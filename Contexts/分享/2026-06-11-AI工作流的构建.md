# Loop Engineering 进化实践：从 prompt 到资料库、Skill、/loop、/workflow

> 作者：王龙祥 · 2026-06-11

---

## 第一部分：我的进化路径

以下是我从只会手写 prompt，到逐步设计出 loop 和 workflow 的真实实践过程。

### 阶段一：发现纯 prompt 开发的问题

最早使用 AI 的方式很简单：遇到问题开一个会话，写 prompt，得到答案，关闭。随后遇到了三个核心痛点：

1. **上下文不可复用。** 每次新会话都要重新交代背景，中断后恢复工作的成本很高。
2. **固定流程的任务反复消耗时间。** 日报每天的步骤完全一样，人工执行没有增量价值。
3. **自查自纠不可靠。** 自己的学习计划，frontmatter 写着「已完成」，但复选框未勾选、笔记未写，在单一会话中根本不会发现。

这三个问题表明：纯 prompt 是点状的、不可持续的。

### 阶段二：准备资料 —— 模板、计划、上下文

要解决这些问题，第一步不是写 loop，而是先把知识库架构搭建好。用 Obsidian 建立了 AI-Work-Kit，核心包含三项：

- **Plans/**：进行中的任务，完成即移走，跨会话可使用 `/resume` 继续；
- **Contexts/**：长期有效的结论和资料，无需每次重新解释；
- **Templates/**：高频任务的格式固化（排查、方案、日报、学习笔记等）。

有了这些，AI 才具备了稳定的「工作记忆」。

### 阶段三：Skill 开发 —— 把高频操作封装成触发词

资料库建立后，每次调用仍需要写较长的指令。于是把模板对应的常用操作封装成 Skill（Cursor / Claude Code / 人类可读三端同步），例如：输入「日报」触发复盘助理，输入「续做」触发续做助手，输入「准备资料」触发 PM 物料整理。

Skill 保证了执行路径的一致性，避免每次临时发挥。至此，大部分日常任务已经比纯 prompt 高效。

### 阶段四：/loop 接管重复流程

有些任务不仅是高频，而且是完全固定的流程，比如日报。步骤始终是：统计 git 提交、读取 Plans 进度、按模板写文件。将其设计成 loop：每晚 21:28 定时自动运行，包含目标、范围、验证、升级、停止五个护栏。日报从此不用手动编写。这是从「交互式」到「无人值守」的一次升级。

### 阶段五：/workflow 处理交叉审计

单会话自查的可靠性不足。学习进度审计需要同时检查 frontmatter、复选框、概念卡、笔记四个维度，人工打开多个文件来回比对效率低，且容易忽略不一致之处。于是设计了一个 workflow，并行启动多个 sub-agent 各自独立取证，最后汇总比对。它发现了此前未意识到的进度偏差——声称完成但证据不匹配。

### 总结进化路径

```
纯 prompt → 资料库（模板/计划/上下文）→ Skill → /loop → /workflow
```

每一步解决上一步无法解决的问题。从「杠杆点在写 prompt」到「杠杆点在设计循环」，在我的实践中就体现为这条进化链。

## 第三部分：作品一 —— /loop 自动日报（阶段四）

> 前置依赖：阶段二的资料库 + 阶段三的 Skill

### 1. 解决的问题

日报。每天步骤固定：统计 git 提交、读取 Plans 进度、按模板写文件。人工执行没有增量价值。

### 2. 进化回溯：没有资料库和 Skill 之前

- 没有模板 → 每天格式可能不一致；
- 没有 Plans → 无法获知当日的进度情况；
- 没有 Skill → 每次需要编写较长的 prompt 来描述日报逻辑。

### 3. 有了资料库和 Skill 之后

- 日报模板固定结构（今日完成 / 明日计划 / 阻塞项）；
- Plans 目录记录当日任务进展；
- 复盘助理 Skill 封装完整日报逻辑（含「仅统计本人 git 提交」等规则）。

### 4. Loop 设计（五个护栏）

| 组件 | 设定 |
|------|------|
| 目标 | 每日 21:28 生成日报，写入 `Contexts/日报/` |
| 范围 | 只统计本人 git 提交（`--author` 过滤），只读本仓库当日 Plans/ |
| 验证 | 产出必须符合日报模板结构 |
| 升级 | 无提交且无进展时中止，提醒人工补充 |
| 停止 | 写入成功即停止 |

### 5. 运行状态

- 日报 loop：每日 21:28，已配置并运行，首份产出见 [[Contexts/日报/2026-06-10]]；
- 周报 loop：每周五 20:00 已配置，自动汇总本周日报，本周五首次运行。

## 第四部分：作品二 —— /workflow 学习进度自审计（阶段五）

> 前置依赖：阶段二的资料库 + 阶段三的 Skill + 阶段四的 loop 思维

### 1. 解决的问题

自我学习进度的交叉审计。学习记录存在「声称状态与真实证据不符」的问题——frontmatter 写「已完成」，但复选框未勾选、笔记未写。单会话自查容易遗漏这种偏差。

### 2. 进化回溯：没有资料库和 Skill 之前

- 没有统一的课程 plan 结构 → 每节课格式不一致，无法统一审计；
- 没有 Plans 目录 → 不清楚有哪些课程需要审计；
- 没有固化的编排脚本 → 每次需要手动描述各 agent 的审计逻辑。

### 3. 有了资料库之后

- 每节课 plan 统一结构：frontmatter 状态 + 步骤复选框 + 资料区双链概念卡 + 笔记沉淀要求；
- `Plans/学习/` 记录课程清单，workflow 自动发现待审计对象；
- 审计逻辑已固化为命名 workflow（`learning-audit`，已入库），下次可一句话复跑。

### 4. Workflow 设计（并行独立取证）

| Agent | 审计维度 | 数据来源 |
|-------|----------|----------|
| A | frontmatter 声称状态 | 每节课 plan 的 `status:` 字段 |
| B | 步骤复选框实际勾选 | 同一文件内的 `- [ ]` / `- [x]` |
| C | 概念卡是否存在 | `Contexts/LLM学习/概念/` |
| D | 学习笔记是否产出 | `Contexts/LLM学习/笔记/` |

每节课一个取证 agent 并行独立采集四类证据，最后由汇总 agent 交叉比对、出报告。审计逻辑与被审计对象分离，结论不依赖单一会话的自我判断。

### 5. 运行数据

- 耗时约 72 秒，5 个 subagent，19 次工具调用；
- 审计范围：4 节课（LLM 与提示词、上下文管理、RAG、Agent）。

### 6. 审计结论（摘自 [[Contexts/LLM学习/笔记/2026-06-11-学习进度审计报告]]）

```text
【一致性检查结果】

1. frontmatter 状态：已完成 3 节、进行中 1 节
2. 复选框勾选：26 个步骤中，勾选数为 0
3. 概念卡：齐备（前置资料已就位，但不能证明学习完成）
4. 学习笔记：4 节课均无对应学习笔记

【结论】声称状态与真实完成度存在系统性偏差，4 节课实际均为「未开始」。
建议暂停新课程，先按步骤补全复选框、练习与笔记。
```

### 7. 为什么这个 workflow 依赖前面的积累

- 如果没有统一的课程 plan 结构，各 agent 无法按同一标准取证；
- 如果没有 Plans 目录，workflow 不知道要审计哪些文件；
- 如果没有固化的命名 workflow，每次启动都要重复描述编排逻辑。

这就是为什么顺序是：**资料库 → Skill → /loop → /workflow**，而不是反过来。

## 第五部分：总结 —— 每一步都是被问题推着走

| 遇到的问题 | 解决方案 | 产物 |
|-----------|----------|------|
| 上下文不持久、格式乱 | 建资料库 | 模板 / Plans / Contexts |
| 高频操作每次重复写 | 封装 Skill | 触发词技能（三端同步） |
| 固定流程人工做浪费 | 定时无人值守 | /loop 日报、周报 |
| 多维度交叉审计自查不可靠 | 并行 agent 比对 | /workflow 学习审计 |

**核心结论**：先有资料库让 AI 有稳定的工作记忆，再有 Skill 固化执行路径，然后 /loop 才能安全地无人值守，最后 /workflow 才能可信地交叉审计。从 prompt 到 loop 的每一步，都建立在前一步的基础之上。
