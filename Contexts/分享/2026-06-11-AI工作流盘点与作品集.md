# AI 工作流实践：问题、构建与作品

> 作者：王龙祥 · 2026-06-11

---

## 一、发现的问题

这套工作流由日常工作中三类实际问题推动形成：

**1. 上下文不可复用。** 排查记录、技术方案、学习笔记分散在会话记录和临时文档中。AI 会话不持久，每次新会话都要重新交代背景；任务中断后再续做，恢复现场的成本很高。

**2. 固定流程的工作在重复消耗时间。** 日报是典型：每天的步骤完全一致——统计当日 git 提交、查看 plan 进展、按模板成文。流程固定、结果可验证，人工执行没有增量价值。周报、PM 物料整理（如收银台参数对照）属于同一类。

**3. 自查自纠不可信。** 有一类检查需要跨多个文件交叉比对证据。例如学习计划标注「已完成」，但标注状态、步骤勾选、笔记产出是否一致，靠自己单会话检查很容易被简化或跳过。

---

## 二、工作流的构建

针对三类问题，分三步搭建，对应 Loop Engineering 的演进路线（写 prompt → 设计 loop → 编排 agent）：

### 第一步：知识库打底（解决问题 1）

建立 AI-Work-Kit 知识库，Obsidian 存储，Cursor 与 Claude Code 接入同一份 Vault。两条核心约定：

- `Plans/` 只放进行中任务，完成即清理；`Contexts/` 沉淀长期有效的结论与资料；
- 高频任务固化为 17 个模板、8 个 Skill（三端同步），以触发词调用，执行路径一致。

由此任务可跨会话续做：`/resume` 读取 plan 即可接续，无需复述背景。

### 第二步：/loop 接管重复流程（解决问题 2）

把日报、周报的目标、触发条件、验证规则、停止条件一次性定义好，交给定时循环执行：

| 任务 | 调度 | 产出 |
|------|------|------|
| 每日日报 | 每晚 21:28 | `Contexts/日报/YYYY-MM-DD.md` |
| 每周周报 | 周五 20:00 | `Contexts/周报/` |

护栏是这一步的设计重点：只统计本人提交（`--author` 过滤）、产出必须符合模板结构、材料不足时中止并要求人工补充。无护栏的循环只会更快地产出错误结果。

### 第三步：/workflow 承担交叉审计（解决问题 3）

对「单会话易漏」的多文件验证任务，用 `/workflow` 并行编排多个 subagent，各自独立取证后汇总比对。审计逻辑与被审计对象分离，结论不依赖单一会话的自觉性。

## 三、产出的作品

### 作品 A（/loop）：自动日报与周报

每晚 21:28 自动统计当日本人 git 提交与 plan 进展，按模板写入 `Contexts/日报/`（示例：[[Contexts/日报/2026-06-10]]）；每周五汇总成周报。已持续产出。

### 作品 B（/workflow）：学习进度审计

并行 4 个 subagent，各审计一节 LLM 学习课，对「frontmatter 声称状态、步骤复选框、概念卡、学习笔记」四个证据源交叉比对，汇总为 [[Contexts/LLM学习/笔记/2026-06-11-学习进度审计报告]]。运行数据：5 个 agent、约 72 秒、19 次工具调用。

**审计发现实质问题**：四节课均声称完成或进行中，但步骤复选框全部未勾选、学习笔记缺失——声称进度与真实进度存在系统性偏差。这正是 workflow 相对单会话的价值所在。

### 其他日常产出（手工层）

- [[Contexts/收银台/MSPay收银台配置对照表]]：新 App 接入收银台的参数对照，PM 可直接据此向收银台部门发起申请；
- `Plans/Bug排查/`、`Plans/学习/` 等模板化 plan 与跨会话续做记录。

## 四、当前全景与演示顺序

| 层级 | 构成 | 适用场景 |
|------|------|----------|
| 手工层 | 8 个 Skill（模板开工、续做、需求分析、功能开发、Figma、PM 物料、日报周报、学习） | 需要人在环中判断 |
| 自动层 | 2 个 /loop（日报、周报） | 流程固定、可验证 |
| 重型层 | /workflow 按需编排 | 多文件交叉审计 |

演示顺序：① 收银台对照表（手工层真实交付物）→ ② `Contexts/日报/` 的 loop 产出与定时任务列表 → ③ 学习审计报告（重点：发现了人工自查未发现的进度偏差）→ ④ 语音读书笔记收尾。
