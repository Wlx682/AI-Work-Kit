# 资料与代码仓库边界

> 原则：**凡是「资料」——可积累、可复盘、可反哺工作流的内容——一律落在 AI-Work-Kit 固定路径；代码仓库只放代码与工程必需文档。**

## 为什么

- Plan、需求分析、对照表、日报周报、踩坑、决策需要**长期可检索、可链接、可续做**。
- 散落在各业务仓库 `docs/` 里无法形成统一工作流；Plan 完成情况也无法反馈到 Skill / 模板优化。
- Obsidian 双向链接、`/resume`、enquire MCP 检索都依赖 Vault 内固定目录。

## 固定位置（全库唯一）

| 类型 | 路径 | 示例 |
|------|------|------|
| 进行中的 Plan | `Plans/【中文分类】/YYYY-MM-DD-标题.md` | `Plans/功能开发/2026-06-11-会员-盒子龙虾切换.md` |
| 长期知识 / PM 物料 | `Contexts/【主题】/` | `Contexts/收银台/MSPay收银台配置对照表.md` |
| 日报 / 周报 | `Contexts/日报/`、`Contexts/周报/` | |
| 决策与约定 | `Contexts/决策/` | 本文 |
| 模板 | `Templates/` | |
| Skill 说明（Vault 副本） | `Skills/` | 与 `~/.cursor/skills/` 同步 |

## 代码仓库里**不放**什么

- 功能开发 Plan、需求分析 Plan、排查 Plan
- PM 对照表、收银台配置表、会议结论（应沉淀到 `Contexts/`）
- 为个人工作流服务的实施方案（与代码提交无直接关系的）

## 代码仓库里**可以**放什么

- `README`、构建说明、架构说明（**团队共享、随 PR 演进**的工程文档）
- 与某次 MR 强绑定的 code review 笔记（若团队约定在仓内）
- 自动生成、工具链要求的文件

> 判断标准：这份文档是「**我的工作流资产**」还是「**这个仓库的运行时说明**」？前者 → Vault；后者 → 可留仓库。

## Plan → 工作流反馈

1. Plan 用 `Templates/` 起稿，按 `Templates/模板约定.md` 命名与 `status`。
2. 做完后：结论沉淀 `Contexts/`（踩坑、决策、对照表）；Plan 可删或标 `已采纳`。
3. 若模板缺章节、Skill 步骤不顺：在 Plan 末尾记「流程反馈」，定期改 `Templates/` 或 `Skills/`。

## 续做（统一入口）

```text
/resume plan=Plans/功能开发/2026-06-11-会员-盒子龙虾切换.md 进度=Phase A
```

Vault 根目录：`~/Documents/AI-Work-Kit`
