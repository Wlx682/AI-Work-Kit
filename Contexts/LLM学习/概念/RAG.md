# 概念：RAG（检索增强生成）

> **学习顺序：第 3 课。** 先掌握 [[大语言模型]] 和 [[提示词工程]]。

## 一句话

**先从你的资料里检索相关片段，再让模型基于片段回答**——避免瞎编，也避免把整个知识库塞进 prompt。

## 流程

```
用户问题 → 检索（关键词/向量/混合）→ Top-K 笔记片段 → 拼进 prompt → 模型生成
```

## 和第 2 课「外置记忆」的关系

| 第 2 课 | 第 3 课 RAG |
|---------|-------------|
| 你知道材料在 `Contexts/`，手动 `@` | 你**不知道**在哪份笔记，用问题去**搜** |
| 你决定塞哪一段 | 检索器返回 Top-K 相关段，再塞进 prompt |

RAG = **外置记忆的自动化取数**。

## 检索方式（知道名字即可）

| 方式 | 特点 |
|------|------|
| 关键词 / BM25 | 精确词匹配，快 |
| 向量 / Embedding | 语义相近也能找到（「超时」≈「请求慢」） |
| 混合 | enquire 的做法：关键词 + 向量 |

## 和「全文粘贴」对比

| | 全文粘贴 | RAG |
|--|----------|-----|
| 成本 | 上下文爆炸 | 只取相关段 |
| 准确性 | 易被无关信息干扰 | 有出处可核对 |
| 更新 | 每次手动贴 | 索引更新即可 |

## 你已经在用的 RAG

- **enquire-mcp**：`obsidian_search` 搜 `Plans/`、`Contexts/`
- 问「知识库里有没有 API 超时记录？」= 典型 RAG 使用场景

详见 [[MCP进阶指南]]。

## RAG 的局限（必知）

- 检索不到 → 模型可能瞎编（要说「知识库未命中」）
- 片段太短 → 断章取义
- 索引旧了 → 新笔记搜不到（要 `index` 更新）
- **不替代 Skill 流程** — 找到 plan 后仍用 `/resume plan=Plans/... 进度=...` 续做
