# 概念：上下文管理（Context Management）

> **学习顺序：第 2 课。** 前置：[[大语言模型]]、[[提示词工程]]。

## 一句话

在**有限的上下文窗口**里，决定「给模型看什么、不看什么、先看什么」——不是越多越好。

## 核心问题

模型一次能看的 token 有限 → 材料一多就：**塞不下 / 变慢变贵 / 无关信息干扰答案**。

## 四种常用策略

| 策略 | 做法 | Kit 中的例子 |
|------|------|--------------|
| **精选** | 只贴与当次任务相关的片段 | plan 里写「已知现象」，不全贴仓库 |
| **摘要** | 长文先压缩成要点再进 prompt | 会议速记 → 纪要再讨论 |
| **分层** | 常驻规则 + 当次任务 + 按需引用 | `.cursorrules` + User 指令 + `@` 单个文件 |
| **外置记忆** | 放不下的存知识库，用时再取 | `Contexts/`、`Plans/`（第 3 课 RAG 深化） |

## 上下文分层（Cursor 里）

```
┌─────────────────────────────────┐
│ System：.cursorrules（常驻）     │
├─────────────────────────────────┤
│ Skill：固化流程（按任务加载）     │
├─────────────────────────────────┤
│ User：当次背景 + 参数            │
├─────────────────────────────────┤
│ 附件：@ 文件 / 选中代码 / 日志片段│
└─────────────────────────────────┘
```

**原则**：下层少而精，上层（User/附件）只加当次需要的。

## 长日志 / 大方案怎么处理

1. **不要**一次贴 500 行 — 先问：错误栈顶 20 行？相关请求那一段？
2. 让 AI **先摘要再分析**：「先列出时间线和异常点，再排查」
3. 仍不够 → 存 `Contexts/` 或 plan，需要时 `@` 或第 3 课的检索

## 和 RAG 的关系（预告第 3 课）

| 上下文管理 | RAG |
|------------|-----|
| 你决定塞什么 | 系统先从库裡**找**相关的再塞 |
| 手动 `@`、摘要 | 自动 search + Top-K 片段 |

## 常见误区

- ❌ 上下文越多越好 — 无关内容会「带偏」模型
- ❌ 把所有规范每次全贴 — 应用 `.cursorrules` / Skill 分层
- ❌ 超长材料硬塞 — 先摘要或检索
